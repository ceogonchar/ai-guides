# 🧠 Память AI-агента в OpenClaw: Полное руководство

> *Ваш агент забывает всё через 5 минут? Эта инструкция решает проблему раз и навсегда.*

## Проблема: AI-амнезия

Каждый раз, когда сессия AI-агента заканчивается, он теряет **весь контекст**. Представьте сотрудника, который каждое утро приходит на работу и не помнит ничего — ни задачи, ни клиентов, ни договорённости. Именно так работают AI-агенты без системы памяти.

OpenClaw решает эту проблему через **файловую систему памяти** — набор markdown-файлов, которые агент читает при старте и обновляет в процессе работы. Никаких баз данных, никаких сложных сервисов — просто текстовые файлы, которые вы можете читать и редактировать сами.

---

## Архитектура: 4 уровня памяти

```
┌─────────────────────────────────────────────┐
│            MEMORY.md (Неокортекс)           │
│     Долгосрочная кураторская память          │
│     Самое важное. < 100 строк.              │
├─────────────────────────────────────────────┤
│     memory/decisions.md + facts.md          │
│     🔒 ИММУННЫЕ — никогда не удаляются      │
│     Решения, контакты, ID, аккаунты         │
├─────────────────────────────────────────────┤
│     memory/weeks/W<N>.md                    │
│     Еженедельные сжатки (воскресенье)       │
│     Ключевые события за неделю              │
├─────────────────────────────────────────────┤
│     memory/YYYY-MM-DD.md (Гиппокамп)       │
│     Дневники — сырые логи каждого дня       │
│     Подробно, но временно                   │
└─────────────────────────────────────────────┘
```

### Уровень 1: MEMORY.md — "Неокортекс"

Главный файл. Агент читает его **при каждом старте сессии**. Здесь только самое критичное:

- Кто вы, ваши проекты, цели
- Текущие критические задачи
- Финансовый snapshot
- Ключевые люди и контакты
- Уроки и правила

**Правило:** MEMORY.md должен быть **компактным** — до 100 строк, до 2000 токенов. Если разрастается — пора дефрагментировать.

### Уровень 2: Дневники — memory/YYYY-MM-DD.md

Каждый день — отдельный файл. Агент записывает сюда всё, что произошло:

```markdown
# 2026-03-18

## 10:30 — Проверил email
- 3 новых письма от клиентов
- Счёт от поставщика на $450

## 14:00 — Встреча с партнёром
⚠️ DECISION: Запускаем новый продукт к 1 апреля
⚠️ LESSON: Всегда записывать договорённости сразу после звонка

## 17:00 — Задачи
📝 LOG: Обновил CRM, добавил 5 контактов
```

### Уровень 3: Иммунные файлы

**decisions.md** — каждое принятое решение. **НИКОГДА не удаляется**, даже при сжатии:

```markdown
## 2026-03-15 — Решение: смена поставщика
Переходим с Provider A на Provider B. Причина: цена на 30% ниже.
Ответственный: менеджер. Дедлайн: 1 апреля.
```

**facts.md** — справочник: люди, ID, аккаунты, инфраструктура. Агент обновляет при каждом новом контакте.

### Уровень 4: Еженедельные сжатки

Каждое воскресенье 7 дневников сжимаются в `memory/weeks/W<N>.md`:
- ✅ Оставить: решения, уроки, ошибки, цифры, новые контакты
- ❌ Удалить: промежуточные шаги, debug-логи, рутинные проверки
- Оригиналы → `memory/archive/`

---

## 🏷️ Обязательные теги в дневниках

Каждая запись должна быть помечена тегом. Это критично для автоматического сжатия:

| Тег | Значение | При сжатии |
|-----|----------|------------|
| `⚠️ DECISION` | Принятое решение | **Никогда не удаляется** → дублируется в decisions.md |
| `⚠️ LESSON` | Урок, инсайт | **Не удаляется** |
| `⚠️ BLOCKER` | Блокер | **Не удаляется** пока не решён |
| `📝 LOG` | Обычный лог | Удаляется при сжатии |

**Правило:** если запись не помечена тегом, она считается `📝 LOG` и будет удалена при следующем сжатии.

## 🧬 Hippocampus Protocol: как агент решает, что помнить

Это самая мощная часть системы. Вместо того чтобы помнить **всё** (дорого и шумно) или забывать **всё** (бесполезно), агент использует формулу **экспоненциального затухания** — как настоящий мозг.

### Формула

```
retention = importance × e^(-λ × age)
```

Где:
- `importance` — базовая важность (0-1)
- `λ` (lambda) — скорость затухания (чем больше, тем быстрее забывается)
- `age` — возраст в "ходах" (turns)

### Таблица скоростей затухания

| Тип сообщения | λ (decay) | Минимум | Пример |
|---------------|-----------|---------|--------|
| Решение | 0.03 | 60% | "Решили: запускаем продукт к 1 апреля" |
| Эмоция пользователя | 0.05 | 50% | "Блин, опять сломалось!" |
| Запрос пользователя | 0.08 | 40% | "Проверь почту" |
| Активная задача | **0.00** | **100%** | Незавершённая задача — НЕ забывается! |
| Завершённая задача | 0.15 | 10% | "Готово, письмо отправлено" |
| Результат инструмента | 0.20 | 5% | Вывод команды, результат поиска |
| Cron-уведомление | 0.40 | 2% | "Heartbeat OK" |
| Болтовня | 0.50 | 1% | "Ок", "Понял", "👍" |

### Кто сказал — имеет значение (Source Weighting)

Не все сообщения равны. Прямое указание от владельца весит больше, чем системное уведомление:

| Источник | Вес |
|----------|-----|
| Владелец (прямое сообщение) | 1.0 |
| Владелец (голосовое) | 1.0 |
| Пересланное сообщение | 0.7 |
| Член команды | 0.6 |
| Внешний человек | 0.5 |
| Результат инструмента | 0.3 |
| Системный cron | 0.2 |

### Эмоциональные модификаторы (Amygdala Layer)

Эмоциональный контекст усиливает или ослабляет запоминание:

| Эмоция | Модификатор | Триггеры |
|--------|-------------|----------|
| Фрустрация | **+30%** | "блин", "опять", "не работает" |
| Срочность | **+25%** | "срочно", "сейчас", "ASAP" |
| Похвала | **+20%** | "молодец", "отлично", "супер" |
| Успех | **+15%** | Задача выполнена |
| Рутина | **-20%** | Heartbeat, статус-проверки |

### Итоговая формула скоринга

```
final_score = base_importance 
            × source_weight 
            × (1 + emotional_modifier) 
            × context_multiplier
```

**Пример:** Владелец пишет голосовое с фрустрацией "Блин, опять клиент не заплатил!"
```
score = 0.9 (user_intent) × 1.0 (voice) × 1.30 (frustrated) × 0.5 (main_session) = 0.585
→ Compressed tier — сохранится как summary
```

### Правила иммунитета (Immunity Rules)

Некоторые вещи агент **не имеет права забывать**, независимо от decay:

| Правило | Decay | Условие |
|---------|-------|---------|
| ❗ Незавершённая задача | **0.0** | Пока не completed |
| ❓ Неотвеченный вопрос | 0.01 | Пока не answered |
| ⏰ Дедлайн < 7 дней | **0.0** | Пока не прошёл |
| 💰 Финансовое | 0.02 | Всегда медленный decay |

---

## Три уровня индексации

В зависимости от score, информация хранится по-разному:

```
Score ≥ 0.65  →  🟢 FULL KEEP    — полностью в контексте
Score 0.25-0.65 → 🟡 COMPRESSED  — только summary  
Score < 0.25  →  🔴 SPARSE POINTER — ссылка на файл
```

Это означает, что агент **никогда не теряет данные** — низкоприоритетные записи просто уходят в файлы и достаются по запросу.

## Циклы консолидации

### Light (каждый ход)
- Пересчёт retention scores
- Обновление sparse index
- Быстро, незаметно для пользователя

### Deep (конец сессии или каждые 50 ходов)
- Сжатие записей с низким retention
- Извлечение решений → дневник
- Обновление связей между записями

### Archival (ночью или по воскресеньям)
- Перенос важного в MEMORY.md
- Очистка устаревших sparse pointers
- Генерация еженедельной сводки
- Обновление контактов

## State.json — "Записка следующей смене"

Файл `memory/consolidation/state.json` — это **handoff-протокол**. Когда агент "засыпает", он оставляет записку для следующей сессии:

```json
{
  "incomplete_tasks": [
    {
      "task": "Отправить отчёт клиенту",
      "priority": "HIGH",
      "project": "consulting",
      "notes": "Черновик готов, ждёт утверждения"
    }
  ],
  "running_services": [
    {
      "name": "API сервер",
      "port": 3000,
      "pm2_name": "my-api"
    }
  ],
  "pending_questions": [
    {
      "question": "Какой бюджет на рекламу в апреле?",
      "deadline": "2026-04-01"
    }
  ],
  "last_session_summary": "Работал над отчётом, осталось добавить графики"
}
```

**При каждом старте** агент читает state.json и продолжает с того места, где остановился.

## Sleep-time Compute: агент "думает" когда idle

Когда агент не занят (heartbeat без задач), он не просто ждёт — он **консолидирует память**:

1. **Извлечение** — из последних сообщений достаёт имена → contacts/, задачи → active/, факты → MEMORY.md
2. **Обновление** — заполняет пробелы в контактах, обновляет статусы задач
3. **Предвычисление** — какие задачи скоро потребуют внимания? какие паттерны в запросах?

## Дефрагментация MEMORY.md

MEMORY.md растёт со временем. Когда он становится слишком большим (>100 строк), запускается **дефрагментация**:

### Процесс
1. **Scan** — подсчёт строк, анализ каждого раздела
2. **Hippocampus Analysis** — расчёт retention score для каждого пункта
3. **Consolidate** — сжатие, удаление завершённого, обновление цифр
4. **Archive** — старые данные → `memory/archive/`

### Пример из реальной дефрагментации

```
## Defrag #31

### SCAN Results
Processed: 3 daily files (1,792 строк)

### High retention (>0.8):
- Критическая задача с дедлайном → retention 1.0
- Финансовый кризис → retention 1.0
- Сделка под угрозой → retention 0.95

### Decay applied (удалено из актива):
- Пошаговые детали транзакций → archived
- Debug-логи → удалены
- Промежуточные итерации → archived
- Рутинные проверки email → удалены
```

**Результат:** 1,792 строки → ~85 строк. Сжатие **21x** без потери критичных данных.

## Anti-Amnesia Protocol: правила записи

### Каждая задача заканчивается записью (checkpoint)

После ЛЮБОЙ значимой работы (>5 минут) — обязательно:

```
1. memory/YYYY-MM-DD.md → что сделано, результат
2. state.json → обновить incomplete_tasks
3. decisions.md → если принято решение
4. facts.md → если появился новый контакт/аккаунт
```

### Триггеры для checkpoint

Если произошло хотя бы ОДНО из этого — **ЗАПИШИ**:
- ✅ Создан/изменён файл или проект
- ✅ Запущен/остановлен процесс
- ✅ Установлен пакет
- ✅ Изменена конфигурация
- ✅ Получена важная информация
- ✅ Принято решение

### ❌ Запрещено
- Завершить сессию без записи в memory
- "Запомнить в голове" — это **ИСЧЕЗНЕТ**
- Субагенту завершиться без записи результата

## 🚀 Практическое руководство: настройка с нуля

### Шаг 1: Создайте структуру папок

```bash
mkdir -p memory/{hippocampus,consolidation,weeks,archive,sparse,compressed}
```

### Шаг 2: Создайте MEMORY.md

```markdown
# MEMORY.md — Долгосрочная память

## 🧠 Обо мне
[Кто я, мои проекты, цели]

## 🔴 Критические задачи
[Что нужно сделать СЕЙЧАС]

## ✅ Недавно завершено
[Последние достижения]

## 👤 Ключевые люди
[Контакты, роли]

## 💡 Уроки
[Что я понял, правила]
```

### Шаг 3: Создайте config.yaml (Hippocampus)

Скопируйте этот файл в `memory/hippocampus/config.yaml`:

```yaml
# Hippocampus Protocol Config

decay_rates:
  decision:           0.03
  user_intent:        0.08
  user_emotion:       0.05
  task_active:        0.00   # Не забывать!
  task_completed:     0.15
  tool_result:        0.20
  cron_notification:  0.40
  ephemeral:          0.50

retention_floors:
  decision:           0.60
  user_intent:        0.40
  task_active:        1.00
  task_completed:     0.10
  tool_result:        0.05
  ephemeral:          0.01

source_weights:
  owner_direct:       1.0
  owner_voice:        1.0
  team_member:        0.6
  system_cron:        0.2
  tool_output:        0.3

immunity_rules:
  - type: "incomplete_task"
    decay_rate: 0.0
  - type: "pending_question"
    decay_rate: 0.01
  - type: "deadline_approaching"
    decay_rate: 0.0
    condition: "days < 7"

consolidation:
  light:
    trigger: "every_turn"
  deep:
    trigger: "session_end OR every_50_turns"
  archival:
    trigger: "weekly_sunday"
```

### Шаг 4: Создайте state.json

```json
{
  "incomplete_tasks": [],
  "running_services": [],
  "pending_questions": [],
  "last_session_summary": ""
}
```

### Шаг 5: Добавьте правила в AGENTS.md

Добавьте в ваш `AGENTS.md` (главный файл инструкций агента):

```markdown
## Memory

- **Daily:** `memory/YYYY-MM-DD.md` — raw logs
- **Long-term:** `MEMORY.md` — curated
- **Decisions:** `memory/decisions.md` — НИКОГДА не удаляются
- **Facts:** `memory/facts.md` — люди, ID, аккаунты

### Теги в дневниках (обязательно!)
- `⚠️ DECISION` — дублировать в decisions.md
- `⚠️ LESSON` — не удаляется при сжатии
- `⚠️ BLOCKER` — не удаляется пока не решён
- `📝 LOG` — удаляется при сжатии

### Anti-Amnesia
После любой работы >5 мин — checkpoint:
1. memory/YYYY-MM-DD.md → что сделано
2. state.json → обновить задачи
3. decisions.md → если решение
```

### Шаг 6: Настройте ежедневную проверку

Каждый день при старте агент должен:

```
1. Прочитать MEMORY.md
2. Прочитать memory/YYYY-MM-DD.md (сегодня + вчера)
3. Прочитать state.json → продолжить незавершённое
4. Проверить decisions.md на актуальность
```

## 💡 Практические советы и "грабли"

### Граблей, на которые мы наступали:

1. **"Запомню в голове"** — НЕТ. Через 10 минут сессия может рестартнуть и всё пропадёт. Файлы > мозг.

2. **MEMORY.md разрастается до 500 строк** — агент тратит 30% контекста просто на чтение памяти. Держите до 100 строк, остальное → дневники и archive.

3. **Субагенты не записывают результат** — запустили 5 субагентов, они сделали работу, но не записали. Результаты потеряны. Решение: в task каждого субагента добавлять "⚠️ Запиши результат в memory/".

4. **Дневники без тегов** — при сжатии удаляется ВСЁ, включая важные решения. Решение: теги обязательны.

5. **Решения в дневниках, но не в decisions.md** — дневник сжали, решение потеряли. Решение: всегда дублировать `⚠️ DECISION` в decisions.md.

6. **State.json не обновляется** — агент "просыпается" и не знает, что делал вчера. Решение: обновлять state.json при каждом checkpoint.

## Итого: почему это работает

| Без системы памяти | С Hippocampus Protocol |
|--------------------|-----------------------|
| Агент забывает всё через 5 минут | Критичное помнит навсегда |
| Контекст забивается мусором | 10-21x сжатие без потерь |
| Незавершённые задачи теряются | Immunity rules — decay = 0 |
| Каждая сессия с нуля | State.json — handoff protocol |
| Все данные одинаково важны | Scoring: кто сказал + эмоция + тип |

**Результат:** агент, который помнит как человек — важное надолго, рутину быстро забывает, а критичное не забывает никогда.
